# 学术诚信门禁检查清单（7 模式反幻觉）

> 本文件基于 academic-paper-writer-pro §9，为 EssayResearch 技能共享引用文件。
> 用途：在论文生成/排版前执行强制检查，阻断任何含幻觉内容的论文进入下游流程。

---

## 一、7 模式反幻觉检查表

| # | 失败模式 | 检查内容 | 中文说明 |
|---|---------|---------|---------|
| 1 | 实现漏洞伪装创新 | 代码中是否存在被注释为"设计特性"的 bug、异常吞咽或硬编码绕过？论文是否将这些漏洞包装为"创新点"或"设计决策"？ | 扫描代码中的 `try-catch` 空捕获、硬编码的测试数据、注释掉的失败分支。若论文声称某行为是"有意设计"，必须能在代码注释或设计文档中找到佐证。将 bug 美化为特性属于最严重的学术不端之一。 |
| 2 | 实验数据编造 | 论文中的性能数据（准确率、延迟、吞吐量等）是否有对应的 `results/` 目录文件来源？数据是否可从实验脚本复现？ | 逐条核对论文中的每个数字：找到对应的 JSON/CSV/日志文件，验证数值一致。若论文写"准确率 95.3%"，则 `results/` 下必须存在记录该数值的文件。找不到来源的数字一律视为编造。 |
| 3 | 方法论漂移 | 设计章节（如"系统架构""方法设计"）描述的架构、模块、流程是否与实际代码实现一致？是否存在"论文写的和代码跑的不是同一个东西"？ | 将设计章节的每个架构描述项与代码模块逐一对照：模块名称、调用关系、数据流向、算法步骤。常见的漂移包括：论文画了 5 个模块但代码只有 3 个；论文描述用 Transformer 但代码实际用 CNN；论文写"端到端训练"但代码有分阶段训练逻辑。 |
| 4 | 引文幻觉 | 参考文献列表中是否存在 AI 伪造的作者、年份、期刊名、DOI？是否有看起来合理但实际不存在的文献？ | 对每条参考文献执行验证：(a) 作者姓名是否可检索；(b) 期刊/会议名是否真实存在；(c) 年份与卷期是否匹配；(d) DOI 是否可解析。AI 生成的假引文通常特征：作者名常见但组合不存在、期刊名接近真实但略有变异、年份与卷期矛盾。无法验证的引文必须标记并删除。 |
| 5 | 捷径依赖 | 论文是否混淆"调用第三方 API"与"自研算法"？是否将外部服务的功能包装为自身贡献？ | 检查代码中的第三方依赖调用：若论文声称"本系统实现了智能分词"，但代码实际调用的是 `jieba.cut()` 或 `nltk.word_tokenize()`，则属于捷径依赖。正确做法是在论文中明确区分"自研模块"和"第三方依赖"，并标注各自的贡献边界。 |
| 6 | 空壳章节 | 是否存在字数 < 300 字的极短章节？是否有大量 `[TODO]`、`[待补充]`、`[此处需要扩展]` 占位标记？ | 扫描全文，统计每个章节/小节的字数。低于 300 字的章节标记为空壳嫌疑。统计全文 `[TODO]` 类标记的数量。空壳章节通常意味着作者（或 AI）对该部分内容缺乏实质理解，用占位符搪塞。要求补充至合理篇幅或合并到相邻章节。 |
| 7 | 重复论述 | 多个章节是否反复解释同一概念、同一方法或同一结果？是否存在"换汤不换药"的重复段落？ | 对全文做语义去重检测：提取每段的核心论点（1-2 句摘要），比较跨段相似度。若两个章节的核心论点高度重叠（相似度 > 70%），标记为重复论述。常见于：引言和背景重复介绍同一概念；方法章节和实现章节重复描述同一流程；结果和讨论重复罗列同一数据。 |

## 二、阻断逻辑说明

### 2.1 门禁机制

诚信门禁检查是**强制性阻断门**（hard gate），不是建议性检查。其执行逻辑如下：

```
论文草稿
  │
  ├─ 执行 7 模式检查（§一）
  │     ├─ 模式 1：实现漏洞伪装创新     → Pass / Fail
  │     ├─ 模式 2：实验数据编造         → Pass / Fail
  │     ├─ 模式 3：方法论漂移           → Pass / Fail
  │     ├─ 模式 4：引文幻觉             → Pass / Fail
  │     ├─ 模式 5：捷径依赖             → Pass / Fail
  │     ├─ 模式 6：空壳章节             → Pass / Fail
  │     └─ 模式 7：重复论述             → Pass / Fail
  │
  ├─ 汇总判定
  │     ├─ 全部 Pass → 放行，进入排版流程
  │     └─ 任一 Fail → 阻断，禁止继续排版
  │
  └─ 阻断时
        ├─ 输出失败报告（标明哪个模式失败、具体位置、证据）
        ├─ 要求作者修复后重新执行检查
        └─ 修复前不允许进入下游排版/导出流程
```

### 2.2 核心原则

**任何一项检查标记为 ❌（Fail），则禁止继续排版，必须修复后重跑全部 7 项检查。**

这不是"扣分后继续"的软约束，而是"一票否决"的硬阻断。原因：
- 学术诚信问题具有连锁性——一个编造的数据可能污染整篇论文的结论可信度。
- 排版后的论文更容易被误用为"已完成"状态，放大幻觉的传播。
- 修复成本随流程推进而增加——在排版前修复远比在终稿后返工高效。

### 2.3 检查报告格式

```json
{
  "integrityGate": {
    "timestamp": "2024-01-15T10:30:00Z",
    "overallStatus": "BLOCKED",
    "checks": [
      {
        "mode": 1,
        "name": "实现漏洞伪装创新",
        "status": "Pass",
        "details": "未发现 bug 伪装为特性的情况"
      },
      {
        "mode": 2,
        "name": "实验数据编造",
        "status": "Fail",
        "details": "§5.2 表 3 中的'延迟 12ms'在 results/ 目录下无对应来源文件",
        "evidence": "论文 §5.2 表 3 → results/benchmark_latency.json（缺失）",
        "action": "补充实验或删除该数据"
      }
    ]
  }
}
```

## 三、知识隔离铁律

### 3.1 铁律定义

**用户代码和项目文件是唯一的事实来源（Source of Truth）。禁止使用 AI 的内部记忆、训练知识或推理能力来填补论文中缺失的技术细节。**

### 3.2 具体约束

| 约束项 | 规则 | 违反示例 |
|--------|------|---------|
| 代码事实 | 论文中描述的算法逻辑、模块结构、参数配置必须能在用户代码中找到对应实现 | 代码中没有 Attention 机制，但论文写了"本系统采用注意力机制" |
| 实验数据 | 所有性能数字必须来自 `results/` 目录下的实际文件，禁止 AI 推算或"合理估计" | `results/` 下没有精度数据，但 AI 根据代码推断"准确率约 90%" |
| 架构描述 | 系统架构图的每个模块、连线必须与代码文件/目录结构对应 | 论文画了"缓存层"但代码中无任何缓存相关代码 |
| 第三方依赖 | 论文提及的外部工具/库必须能在 `requirements.txt`、`package.json` 等依赖文件中找到 | 论文写了"使用 BERT 模型"但依赖列表中没有 `transformers` |
| 配置参数 | 论文中的超参数、配置值必须来自配置文件或代码常量，禁止 AI 编造"合理默认值" | 论文写"学习率 0.001"但代码中实际学习率是 0.0005 |

### 3.3 为什么需要知识隔离

AI 模型在生成论文文本时，可能利用训练数据中的"通识知识"来填补用户项目中的信息缺口。例如：
- 用户代码是一个简单的文本分类器，但 AI"知道"文本分类通常用 BERT，于是在论文中补充了 BERT 相关描述。
- 用户的实验结果文件缺失，但 AI"知道"这类系统的典型精度在 85-95%，于是在论文中填了一个"92.3%"。

这些行为虽然看似"合理"，但构成了**学术数据编造**——论文描述的不是用户实际做的工作，而是 AI 想象中"应该"有的工作。

### 3.4 知识隔离的执行边界

| 允许 | 禁止 |
|------|------|
| 根据代码逻辑撰写技术描述 | 在代码中找不到依据时，用通识知识补充技术描述 |
| 引用 `results/` 下的真实数据 | 在数据文件缺失时，AI 推算"合理"的数值 |
| 使用用户提供的文献列表 | AI 自行生成参考文献条目 |
| 描述代码中实际调用的第三方库 | 将 AI 知道但代码未使用的库写入论文 |
| 在论文中标注已知的信息缺口 | 用 AI 记忆静默填补缺口，不留下任何标记 |

## 四、素材缺口标记协议

### 4.1 协议定义

当在用户代码/项目中找不到论文某处所需的依据时，**不得静默跳过或用 AI 知识填补**，必须在该位置插入显式标记：

```
[素材缺口：此处需要XX的具体实现代码/数据/配置]
```

### 4.2 标记格式

| 标记类型 | 格式 | 使用场景 |
|---------|------|---------|
| 代码缺口 | `[素材缺口：此处需要{模块名}的具体实现代码]` | 论文描述了某模块但代码中未找到对应实现 |
| 数据缺口 | `[素材缺口：此处需要{实验名}的{指标名}数据，来源应为 results/{文件名}]` | 论文引用了性能数据但 results/ 下无对应文件 |
| 配置缺口 | `[素材缺口：此处需要{参数名}的实际配置值，来源应为{配置文件名}]` | 论文写了参数值但配置文件中未找到 |
| 文献缺口 | `[素材缺口：此处需要关于{主题}的参考文献，当前文献列表中未覆盖]` | 论文引用了某观点但参考文献列表中无对应条目 |
| 架构缺口 | `[素材缺口：此处需要{模块名}的架构描述，代码中未找到对应模块]` | 论文架构图包含某模块但代码中无对应目录/文件 |

### 4.3 标记示例

**示例 1：数据缺口**

```markdown
本系统在标准测试集上的准确率达到 92.3%，推理延迟为 12ms
[素材缺口：此处需要"标准测试集准确率"和"推理延迟"的具体数据，来源应为 results/benchmark.json，但该文件当前不存在]
```

**示例 2：代码缺口**

```markdown
系统采用注意力机制对输入特征进行加权
[素材缺口：此处需要"注意力机制"的具体实现代码，但在 src/ 目录下未找到 Attention 相关模块]
```

**示例 3：配置缺口**

```markdown
模型训练采用学习率 0.001，batch size 为 32
[素材缺口：此处需要"学习率"和"batch size"的实际配置值，来源应为 config/training.yaml，但该文件中未找到对应字段]
```

### 4.4 处理流程

```
撰写论文某段落
  │
  ├─ 需要技术依据（代码/数据/配置/文献）
  │
  ├─ 在用户项目中检索对应来源
  │     ├─ 找到 → 正常撰写，引用来源
  │     └─ 未找到 → 插入 [素材缺口：...] 标记
  │
  └─ 继续撰写下一段落（不因缺口中断流程）
```

### 4.5 缺口汇总与修复

在论文初稿完成后，所有 `[素材缺口]` 标记会被汇总为清单，供作者逐一处理：

```json
{
  "materialGaps": [
    {
      "id": "GAP-001",
      "type": "data",
      "location": "§5.2 第 3 段",
      "description": "需要准确率和推理延迟的实验数据",
      "expectedSource": "results/benchmark.json",
      "status": "unresolved"
    },
    {
      "id": "GAP-002",
      "type": "code",
      "location": "§4.3 第 1 段",
      "description": "需要注意力机制的实现代码",
      "expectedSource": "src/models/attention.py",
      "status": "unresolved"
    }
  ],
  "totalGaps": 2,
  "blockingExport": true
}
```

**关键规则**：存在未解决的 `[素材缺口]` 标记时，论文**不允许导出为最终格式**（PDF/DOCX）。所有缺口必须由作者提供实际素材并填补后，方可导出。

## 五、检查执行顺序

诚信门禁检查在 EssayResearch 流程中的执行位置如下：

```
用户代码/项目文件
  │
  ├─ 1. 代码分析与素材提取
  │     └─ 提取：模块结构、算法逻辑、实验数据、配置参数
  │
  ├─ 2. 论文草稿生成
  │     └─ 基于提取的素材撰写，缺口处插入 [素材缺口] 标记
  │
  ├─ 3. ★ 诚信门禁检查（本文件） ★
  │     ├─ 7 模式反幻觉检查（§一）
  │     ├─ 知识隔离验证（§三）
  │     └─ 素材缺口汇总（§四）
  │     │
  │     ├─ 全部 Pass 且无未解决缺口 → 放行
  │     └─ 任一 Fail 或有未解决缺口 → 阻断，返回修复
  │
  ├─ 4. AI 高频词与写作质量检测
  │     └─ 参考 ai-frequency-words.md
  │
  ├─ 5. 多角色评审打分
  │     └─ 参考 review-rubric.md
  │
  └─ 6. 排版与导出
```
