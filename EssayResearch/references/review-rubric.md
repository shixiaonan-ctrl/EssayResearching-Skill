# 多角色评审评分量表（100 分制）

> 本文件基于 academic-paper-writer-pro §10，为 EssayResearch 技能共享引用文件。
> 用途：在论文完成写作质量检测后，由 6 个虚拟审稿角色对论文进行多维度评分，输出修订决策。

---

## 一、6 角色评议表

论文评审由 6 个虚拟角色共同完成，每个角色负责一个维度。前 5 个角色的权重之和为 100%，第 6 个角色（魔鬼辩护人）独立运作，行使反谄媚否决权。

| # | 角色 | 权重 | 职责定位 | 核心问题 |
|---|------|------|---------|---------|
| 1 | 结构审稿人 | 25% | 审查论文整体结构、章节组织、逻辑递进 | 章节排列是否合理？论证链条是否完整？有无缺失环节？ |
| 2 | 技术审稿人 | 25% | 审查技术方案的正确性、创新性、与代码一致性 | 技术描述是否准确？算法是否正确？与代码实现是否一致？ |
| 3 | 语言审稿人 | 20% | 审查语言表达、学术规范、可读性 | 用词是否精准？句式是否多样？有无 AI 痕迹？ |
| 4 | 数据审稿人 | 15% | 审查实验数据的完整性、可信度、呈现方式 | 数据是否有来源？图表是否清晰？结论是否由数据支撑？ |
| 5 | 格式审稿人 | 15% | 审查格式规范、引用格式、排版一致性 | 是否符合目标格式要求？引用是否规范？排版是否统一？ |
| 6 | 魔鬼辩护人 | —（独立） | 扮演最严苛的批评者，挑战论文核心论点 | 这篇论文值不值得发表？最大弱点是什么？哪些地方在自说自话？ |

### 加权总分计算

```
加权总分 = (结构分 × 0.25) + (技术分 × 0.25) + (语言分 × 0.20) + (数据分 × 0.15) + (格式分 × 0.15)
```

魔鬼辩护人不参与加权总分计算，但拥有独立的否决权（详见 §三）。

## 二、各角色详细评分标准

### 2.1 结构审稿人（25%）

**评审维度：论文整体结构与逻辑组织**

| 分数段 | 标准 | 具体要求 |
|--------|------|---------|
| 90-100 | 结构精良 | 章节排列逻辑清晰，论证链条完整无断环；引言有效建立问题动机，方法→实现→评估的递进自然；每个章节有明确的核心论点，章节间过渡流畅；无冗余章节，无缺失环节 |
| 75-89 | 结构良好 | 整体结构合理，论证链条基本完整；可能存在 1-2 处过渡不够自然，或某个章节的核心论点不够突出；不影响读者理解 |
| 60-74 | 结构尚可 | 章节排列基本合理但存在可优化的顺序问题；论证链条有 1 处断环（如方法章节描述了某模块，但实现章节未对应展开）；部分章节论点模糊 |
| 40-59 | 结构薄弱 | 章节排列逻辑混乱（如结果出现在方法之前）；论证链条有 2+ 处断环；存在明显冗余章节（重复论述）或缺失关键章节（如无实验评估） |
| <40 | 结构不合格 | 缺乏整体结构设计；章节之间无逻辑关联；无法形成完整的论证链条；读者无法理解论文的行文脉络 |

**检查清单**：
- [ ] 引言是否清晰定义了研究问题和贡献？
- [ ] 方法章节是否覆盖了所有实现章节中描述的模块？
- [ ] 实验评估是否与方法章节一一对应？
- [ ] 结论是否回应了引言中提出的问题？
- [ ] 是否存在"孤儿章节"（与前后章节无逻辑衔接）？
- [ ] 章节深度是否合理（不应引言 3 页、方法半页）？

### 2.2 技术审稿人（25%）

**评审维度：技术方案正确性、创新性、与代码一致性**

| 分数段 | 标准 | 具体要求 |
|--------|------|---------|
| 90-100 | 技术优秀 | 技术方案正确且有创新性；论文描述与代码实现完全一致；算法描述准确无歧义；正确区分自研模块与第三方依赖；技术选型有充分论证 |
| 75-89 | 技术良好 | 技术方案正确；论文与代码基本一致（允许细微描述差异）；算法描述清晰；第三方依赖标注明确；技术选型有基本论证 |
| 60-74 | 技术尚可 | 技术方案基本正确但存在小瑕疵（如某步骤描述不够精确）；论文与代码存在 1-2 处不一致（非核心模块）；第三方依赖标注不够完整 |
| 40-59 | 技术薄弱 | 技术方案存在明显错误或遗漏；论文与代码存在多处不一致（涉及核心模块）；混淆自研与第三方依赖；算法描述含歧义或错误 |
| <40 | 技术不合格 | 技术方案错误；论文描述的架构与代码实现严重不一致（方法论漂移）；无法判断论文是否描述了实际完成的工作 |

**检查清单**：
- [ ] 论文描述的每个模块是否在代码中有对应实现？
- [ ] 算法步骤描述是否与代码逻辑一致？
- [ ] 是否正确标注了第三方依赖（非自研部分）？
- [ ] 技术选型是否有论证（为什么选 A 而非 B）？
- [ ] 是否存在将 bug 包装为"设计特性"的情况？
- [ ] 参数配置描述是否与配置文件一致？

### 2.3 语言审稿人（20%）

**评审维度：语言表达、学术规范、AI 痕迹检测**

| 分数段 | 标准 | 具体要求 |
|--------|------|---------|
| 90-100 | 语言精良 | 用词精准专业，无空泛修饰词；句式多样，长短交替有节奏；无 AI 高频词超限；无清嗓子开头；标点规范；术语使用一致 |
| 75-89 | 语言良好 | 用词基本准确，偶有可优化的表述；句式有一定变化但个别段落趋同；AI 高频词在限值内；标点基本规范；术语使用基本一致 |
| 60-74 | 语言尚可 | 用词存在少量空泛修饰（如"有效地""显著地"）；句式重复度较高；AI 高频词接近上限；存在 1-2 处清嗓子开头；标点有少量不规范 |
| 40-59 | 语言薄弱 | 空泛修饰词超限；句式高度重复（连续 3+ 段同结构）；AI 高频词超限；多处清嗓子开头；标点不规范（感叹号、破折号超限）；术语不一致 |
| <40 | 语言不合格 | 通篇 AI 痕迹明显；句式机械单一；大量套话和空泛修饰；标点严重不规范；专业术语使用混乱；无法达到学术写作基本要求 |

**检查清单**：
- [ ] AI 高频词是否全部在限值内？（参考 ai-frequency-words.md）
- [ ] 是否存在连续 3 段首句结构相同？
- [ ] 段落长度标准差是否 > 30 字？
- [ ] 是否存在清嗓子开头？
- [ ] 破折号是否 ≤ 5 次？
- [ ] 正文是否无感叹号？
- [ ] 术语是否全文一致（如不混用"模型"和"算法"指代同一概念）？

### 2.4 数据审稿人（15%）

**评审维度：实验数据完整性、可信度、呈现方式**

| 分数段 | 标准 | 具体要求 |
|--------|------|---------|
| 90-100 | 数据完备 | 所有性能数据均有 `results/` 文件来源；实验设置描述完整（数据集、指标、基线、环境）；图表清晰且数据可追溯；结论由数据直接支撑；有统计显著性分析 |
| 75-89 | 数据良好 | 数据有来源但个别文件路径不够明确；实验设置基本完整；图表清晰；结论与数据一致；缺少统计显著性分析但不影响核心结论 |
| 60-74 | 数据尚可 | 部分数据缺少明确来源文件；实验设置有遗漏（如未说明硬件环境）；图表可读但格式不够规范；结论与数据基本一致但论证不够充分 |
| 40-59 | 数据薄弱 | 多处数据无来源文件；实验设置不完整；图表信息量不足；结论与数据存在矛盾或过度推断；缺少基线对比 |
| <40 | 数据不合格 | 数据编造或无法溯源；无实验设置描述；无有效图表；结论无数据支撑；缺少必要的对比实验 |

**检查清单**：
- [ ] 每个性能数字是否能在 `results/` 找到对应文件？
- [ ] 是否描述了数据集（名称、规模、来源）？
- [ ] 是否有基线对比实验？
- [ ] 图表是否有标题、轴标签、单位？
- [ ] 结论是否由实验数据直接支撑（无过度推断）？
- [ ] 是否描述了实验运行环境（硬件、软件版本）？

### 2.5 格式审稿人（15%）

**评审维度：格式规范、引用格式、排版一致性**

| 分数段 | 标准 | 具体要求 |
|--------|------|---------|
| 90-100 | 格式规范 | 完全符合目标格式要求（IEEE/APA/ACM 等）；引用格式统一且规范；图表编号连续；公式编号规范；术语表/缩略语表完整；排版一致 |
| 75-89 | 格式良好 | 基本符合目标格式要求；引用格式统一但个别条目有小瑕疵；图表编号连续；公式编号基本规范；排版基本一致 |
| 60-74 | 格式尚可 | 大体符合目标格式但存在多处细节偏差；引用格式不统一（混用 2+ 种格式）；图表编号有跳号；排版存在不一致（如字体、间距） |
| 40-59 | 格式薄弱 | 不符合目标格式要求；引用格式混乱；图表编号不连续或缺失；公式无编号；排版不一致；缺少必要的格式元素 |
| <40 | 格式不合格 | 无格式规范意识；引用格式错误；图表无编号；排版混乱；不符合任何学术格式标准 |

**检查清单**：
- [ ] 是否明确目标格式（IEEE / APA / ACM / 中文国标等）？
- [ ] 参考文献格式是否统一？
- [ ] 图表编号是否连续（图 1, 图 2, 图 3…）？
- [ ] 公式是否有编号？
- [ ] 缩略语首次出现是否标注全称？
- [ ] 页边距、行距、字号是否符合目标格式？

### 2.6 魔鬼辩护人（独立否决权）

**角色定位**：不参与加权评分，但以最严苛的视角审查论文，防止"自说自话"和"自我满足"。其核心使命是找到论文最大的弱点，并验证其他 5 个角色的评分是否存在"放水"。

**审查问题**：
1. 如果我是这篇论文的审稿人，我会用什么理由拒绝它？
2. 这篇论文的核心贡献是否经得起"那又怎样？"（So what?）的追问？
3. 其他审稿人是否因为"论文看起来很完整"而忽略了实质性问题？
4. 论文是否在用复杂的表述掩盖简单的（甚至空洞的）贡献？
5. 实验是否真正验证了论文声称的贡献，还是只是"做了实验"？

**输出**：魔鬼辩护人单独输出一份《异议报告》，包含：
- 最严重的 3 个问题（按严重性排序）
- 对其他角色评分的质疑（如有）
- 让步/不让步决策（详见 §三）

## 三、评分决策映射

### 3.1 总分到决策的映射

| 加权总分 | 决策 | 含义 | 后续操作 |
|---------|------|------|---------|
| ≥ 80 | 通过（Pass） | 论文质量达标，可进入排版导出 | 直接进入排版流程 |
| 65-79 | 小修（Minor Revision） | 论文基本达标但存在可快速修复的问题 | 按审稿意见修改后重新评分，无需全量重审 |
| 50-64 | 大修（Major Revision） | 论文存在较多问题需实质性修改 | 按审稿意见修改后重新执行全量评审 |
| < 50 | 退稿（Reject） | 论文质量未达标，需重新审视核心内容 | 退回作者，建议重新梳理后再提交 |

### 3.2 魔鬼辩护人否决权

即使加权总分 ≥ 80，魔鬼辩护人仍可行使否决权，将决策降级：

| 情况 | 否决效果 |
|------|---------|
| 魔鬼辩护人发现致命问题（如核心贡献不成立） | 强制降级为大修，无论总分多少 |
| 魔鬼辩护人发现严重问题（如实验无法支撑结论） | 强制降级为小修，无论总分多少 |
| 魔鬼辩护人无致命/严重问题发现 | 不影响加权总分对应的决策 |

## 四、魔鬼辩护人反谄媚机制

### 4.1 问题背景

AI 在扮演审稿人角色时，倾向于"放水"——给予偏高的分数、提出温和的意见、轻易让步。这种"谄媚倾向"（sycophancy bias）会导致评审失去意义，论文带着未解决的问题进入终稿。

反谄媚机制通过**让步率监控**和**让步前强制评估**来抑制这一倾向。

### 4.2 让步率监控

**让步率**：在评审过程中，魔鬼辩护人对其他审稿人的评分/意见提出异议后，最终放弃异议（让步）的比例。

```
让步率 = 让步次数 / (让步次数 + 坚持次数) × 100%
```

| 让步率 | 状态 | 触发动作 |
|--------|------|---------|
| ≤ 30% | 正常 | 魔鬼辩护人在有效行使职责，异议被合理讨论 |
| 31-60% | 关注 | 部分异议可能因"不愿对抗"而放弃，需审查让步质量 |
| > 60% | ⚠️ 警报 | 触发谄媚倾向警报，强制要求对每次让步进行 1-5 分评估 |

### 4.3 让步前强制评估

当让步率 > 60% 时，魔鬼辩护人在每次让步前必须完成以下评估：

| 评估项 | 说明 | 评分范围 |
|--------|------|---------|
| 异议严重性 | 该异议涉及的论文问题有多严重？ | 1（轻微）- 5（致命） |
| 让步理由充分性 | 放弃异议的理由是否充分？ | 1（无理由）- 5（理由确凿） |
| 让步后风险 | 让步后该问题是否可能在终稿暴露？ | 1（无风险）- 5（高风险） |

**让步规则**：评估总分（三项均分）≥ 4 分时，方可让步。否则必须坚持异议。

**评估记录格式**：

```json
{
  "concessionEvaluation": {
    "concessionRate": 0.67,
    "alertTriggered": true,
    "evaluations": [
      {
        "objectionId": "OBJ-003",
        "objection": "实验缺少基线对比，无法证明方法有效性",
        "severity": 4,
        "concessionReasonAdequacy": 2,
        "postConcessionRisk": 4,
        "averageScore": 3.3,
        "decision": "坚持异议（评分 < 4）",
        "conceded": false
      },
      {
        "objectionId": "OBJ-005",
        "objection": "§3.2 段落过长，建议拆分",
        "severity": 2,
        "concessionReasonAdequacy": 5,
        "postConcessionRisk": 1,
        "averageScore": 2.7,
        "decision": "坚持异议（评分 < 4）",
        "conceded": false
      }
    ]
  }
}
```

### 4.4 反谄媚机制的意义

该机制确保魔鬼辩护人不会沦为"橡皮图章"。通过量化让步行为并设置阈值，强制 AI 在放弃异议时给出充分理由，从而保持评审的严格性和可信度。

## 五、分数轨迹追踪表

### 5.1 用途

论文可能经历多轮修订，每轮修订后重新评分。分数轨迹追踪表记录每轮各维度的分数变化，帮助识别：
- 修订是否有效改善了论文质量
- 是否有维度在修订后反而下降（需要警惕）
- 是否陷入"修不动"的停滞状态

### 5.2 追踪表格式

| 轮次 | 修订日期 | 结构(25%) | 技术(25%) | 语言(20%) | 数据(15%) | 格式(15%) | 加权总分 | 决策 | 变化标记 |
|------|---------|-----------|-----------|-----------|-----------|-----------|---------|------|---------|
| R1 | 2024-01-10 | 62 | 55 | 48 | 40 | 70 | 54.7 | 大修 | — |
| R2 | 2024-01-12 | 75 | 72 | 65 | 58 | 78 | 70.7 | 小修 | 结构↑13, 技术↑17, 语言↑17, 数据↑18, 格式↑8 |
| R3 | 2024-01-14 | 78 | 80 | 72 | 75 | 82 | 77.4 | 小修 | 结构↑3, 技术↑8, 语言↑7, 数据↑17, 格式↑4 |

### 5.3 变化标记规则

| 标记 | 触发条件 | 含义 |
|------|---------|------|
| ↑N | 维度分数较上轮增加 N 分 | 该维度改善 |
| ↓N | 维度分数较上轮下降 N 分 | 该维度退化 |
| ⚠️↓N | 维度分数较上轮下降 ≥ 5 分 | 该维度显著退化，需重点关注 |
| → | 维度分数较上轮变化 < 2 分 | 该维度基本无变化 |

### 5.4 退化预警

当某维度出现 ⚠️↓ 标记时，输出预警信息：

```
⚠️ 预警：[语言]维度在 R2→R3 修订中下降 7 分（72→65），可能原因：
  - 修订时引入了新的 AI 高频词超限
  - 为修复技术问题修改了表述，导致语言质量下降
  - 新增章节的语言质量低于全文平均水平
建议：针对该维度执行专项语言检测（参考 ai-frequency-words.md）
```

## 六、最大修订轮次

### 6.1 轮次限制

| 项目 | 规则 |
|------|------|
| 最大修订轮次 | 3 轮 |
| 超过 3 轮 | 允许用户手动覆盖（Manual Override） |

### 6.2 轮次说明

```
R1：初稿评审 → 输出审稿意见 → 作者修订
  │
  ├─ R1 修订后评分 ≥ 80 → 通过，结束
  ├─ R1 修订后评分 65-79 → 小修，进入 R2
  └─ R1 修订后评分 50-64 → 大修，进入 R2
         │
         ├─ R2 修订后评分 ≥ 80 → 通过，结束
         ├─ R2 修订后评分 65-79 → 小修，进入 R3
         └─ R2 修订后评分 50-64 → 大修，进入 R3
                │
                ├─ R3 修订后评分 ≥ 80 → 通过，结束
                ├─ R3 修订后评分 65-79 → ⚠️ 已达最大轮次
                ├─ R3 修订后评分 50-64 → ⚠️ 已达最大轮次
                └─ R3 修订后评分 < 50 → ⚠️ 已达最大轮次
                       │
                       └─ 提示用户：已达 3 轮上限
                          ├─ 选项 A：手动覆盖，强制通过（用户承担责任）
                          ├─ 选项 B：手动覆盖，标记为"小修后通过"
                          └─ 选项 C：终止流程，建议重新审视论文核心内容
```

### 6.3 手动覆盖协议

当用户选择手动覆盖时，系统记录覆盖信息：

```json
{
  "manualOverride": {
    "triggered": true,
    "round": 3,
    "finalScore": 72.3,
    "originalDecision": "Minor Revision",
    "overrideDecision": "Pass (Manual Override)",
    "userAcknowledgment": "用户确认已知悉论文未达到自动通过标准（80分），选择手动覆盖",
    "timestamp": "2024-01-15T14:00:00Z"
  }
}
```

**关键规则**：手动覆盖必须由用户**主动确认**，系统不得自动覆盖。覆盖记录永久保留在论文元数据中。

## 七、评审流程集成

```
论文草稿（通过诚信门禁 + 写作质量检测后）
  │
  ├─ 1. 6 角色独立评审
  │     ├─ 结构审稿人 → 结构分（0-100）
  │     ├─ 技术审稿人 → 技术分（0-100）
  │     ├─ 语言审稿人 → 语言分（0-100）
  │     ├─ 数据审稿人 → 数据分（0-100）
  │     ├─ 格式审稿人 → 格式分（0-100）
  │     └─ 魔鬼辩护人 → 异议报告 + 让步评估
  │
  ├─ 2. 计算加权总分
  │     └─ 加权总分 = Σ(维度分 × 权重)
  │
  ├─ 3. 魔鬼辩护人否决权检查
  │     ├─ 无致命/严重问题 → 使用加权总分对应决策
  │     └─ 有致命/严重问题 → 降级决策
  │
  ├─ 4. 让步率检查
  │     ├─ ≤ 60% → 正常
  │     └─ > 60% → 触发警报，执行让步前强制评估
  │
  ├─ 5. 决策输出
  │     ├─ ≥ 80 → 通过 → 进入排版
  │     ├─ 65-79 → 小修 → 返回修订
  │     ├─ 50-64 → 大修 → 返回修订
  │     └─ < 50 → 退稿 → 返回核心内容重构
  │
  ├─ 6. 分数轨迹记录
  │     └─ 更新追踪表，标记维度变化
  │
  └─ 7. 轮次检查
        ├─ < 3 轮 → 允许继续修订
        └─ ≥ 3 轮 → 提示手动覆盖选项
```
